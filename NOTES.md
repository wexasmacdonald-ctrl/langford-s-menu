SYSTEM PLAN (Menu Boards)
-----
Objective

Run 3 TV menu boards (optionally expandable to 4 later) using one onn stick per TV. Each stick loads a local URL from a local server on the store network. The system must be readable, consistent, easy to update, and recover on its own after power cycles.

1) System Architecture
Components

3 TVs mounted in the store (standard 16:9, typically 43–55").

3 onn sticks (one per TV) used only as “display clients.”

Local Wi-Fi router providing the LAN (internet optional).

Local server device (Windows PC / Chromebook / etc.) hosting the menu application and acting as the “brain.”

How it behaves

Each stick opens a dedicated URL:

Screen 1: /s1

Screen 2: /s2

Screen 3: /s3

The server decides what content should be shown (daily special logic, daypart logic, slide sets, etc.) based on the server device time.

The sticks are not responsible for scheduling. They only show whatever the URL renders.

2) Network Requirements (to prevent random breakage)

The server must have a stable IP on the LAN (DHCP reservation).

Use a stable address scheme:

Either a fixed IP (http://192.168.x.x/s1)

Or a local hostname (http://menu.local/s1) if the router supports it.

The goal is: the sticks’ saved URLs never have to change.

3) “No Support Calls” Operational Requirements
Auto-recovery after power loss

If power returns and the router/server/TVs boot, the menu should come back with minimal staff action:

Router comes up → LAN exists

Server starts hosting automatically

Sticks launch their URLs automatically

Device behavior

onn sticks configured in kiosk-style behavior:

auto-launch assigned URL on boot

no sleep / screensaver

stable power (ideally wall adapter if TV USB is weak)

Server device must:

run during business hours without manual steps

have correct timezone set (America/Montreal)

keep accurate time

4) Screen Strategy (3 TVs with “5 slides total”)
Core rule

Rotation is used only to avoid shrinking text. We do not cram everything onto one screen at once.

Screen assignments

TV 1 — Daily Specials (static or minimal rotation)

Must feel like the “hero” screen.

Always visible daily special + “includes 355ml beverage.”

Also includes Wed/Thu/Fri pizza promo banner and “Everyday 9” pizza of choice.”

TV 2 — Hot Food Board (rotates A/B)

Slide A: Finger foods + Fries

Slide B: Poutine + Fried chicken + Burgers

TV 3 — Pizza Board (rotates A/B)

Slide A: Pizza size/price grid

Slide B: Add-ons + feature pizza list (meats/canadian/classic/greek, etc.)

Rotation timing

Use 12–15 seconds per slide.

Use crossfade only.

Avoid rotating all TVs simultaneously; keep specials steady and rotate the other two.

5) Scheduling Logic
Dayparts

Breakfast / Lunch / Dinner switching exists as a “mode” (if needed for any screen).

Daypart decisions are made from the server clock.

The server renders the correct content for the current daypart.

Daily special logic

Daily specials switch based on weekday.

Server decides what “Today” is.

6) Update & Content Management
Single source of truth

Menu content stored centrally on the server in a simple structured format (e.g., JSON).

Editing prices/items should not require redesigning layouts.

Update behavior

When menu data changes, screens should reflect updates without manual intervention.

Avoid full-page reload loops for normal rotation; rotation should be internal and smooth.

If the network briefly glitches, the screen should not flicker constantly.

7) Stability / Performance Rules
Avoid “reload every 10 seconds”

Rotation should not depend on forcing the stick/browser to constantly reload the URL.

Keep the page steady and update content inside the page to avoid flicker and reduce crash risk on low-power sticks.

Deterministic timing (prevents drift)

When rotating between A/B slides, the displayed slide should be determined by time-based logic (not “toggle state”) so it self-corrects after lag.

If a delay happens, it snaps back to the correct slide on the next tick.

8) Visual Theme System (consistent across all pages)
Theme goal

Match the existing printed menu vibe (orange bar energy, comfort food) but modernize for TV readability.

Theme rules

Dark charcoal backgrounds (no busy photo backgrounds behind text).

Orange/amber for category header bars.

Brick-red accent only for specials/promos.

Large category headers; large item text; prices right-aligned and bold.

Bilingual formatting consistent: French primary, English secondary smaller and muted.

Footer text on every screen: “TAXES EN SUS / TAXES EXTRA”.

Layout rules (so it doesn’t look cramped)

Card-based grid layout, consistent spacing.

Maximum rows per slide tuned so it reads from distance (don’t shrink type to fit).

9) Expansion Path (optional 4th TV later)

If specials become bigger than expected or they want promo focus:

Add a 4th TV as Specials-only and simplify the other boards further.

Architecture stays identical: another stick hits /specials (or /s4).

10) Success Criteria

Readable from typical ordering distance.

Specials feel “big deal” and are always visible.

No manual daily switching needed.

After power loss: system returns with minimal/no staff work.

Consistent theme across all screens as new pages are added.
